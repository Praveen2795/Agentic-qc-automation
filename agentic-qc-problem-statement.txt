================================================================================
AGENTIC QC AUTOMATION — PROBLEM STATEMENT & CONTEXT
================================================================================

This document describes the problem space, constraints, edge cases, and domain
context for automating Quality Control (QC) processes at a large financial
institution. It is intended to be given as context to an AI agent or developer
who needs to understand the problem deeply before designing or building anything.

NO SOLUTION DESIGN IS INCLUDED. This is purely problem + constraints.

================================================================================
1. WHAT IS QC?
================================================================================

Quality Control (QC) at this financial institution is the process of verifying
that business operations (loan servicing, collections, fraud detection, credit
card management, recovery, etc.) were executed correctly — in compliance with
internal policy, regulatory requirements, and business rules.

Each QC process has a corresponding Policy & Procedure (PnP) document that
defines:
- What to check (the validation steps)
- Which systems to pull data from
- What constitutes a pass vs. fail
- What tolerances or thresholds apply
- What timelines are acceptable

QC is performed on a sample of accounts each month. For each account in the
sample, the QC analyst executes every validation step defined in the PnP and
records the result (pass/fail) with evidence in an Excel report.

================================================================================
2. SCALE OF THE PROBLEM
================================================================================

- 300+ distinct QC processes across the organization
- Spanning multiple business units: credit cards, collections, fraud, recovery,
  loan servicing, and others
- Each QC process has its own PnP document
- QC is performed monthly (some quarterly)
- Each monthly QC run covers a sample of accounts (sample sizes vary)
- Each account in the sample requires checking against multiple validation steps
  (anywhere from 5 to 50+ steps per QC)
- Results are compiled into structured Excel reports for review and sign-off

================================================================================
3. HOW QC IS DONE TODAY (MANUAL PROCESS)
================================================================================

Today, QC analysts manually execute each check:

1. Read the PnP document to understand the validation steps
2. Pull data from source systems (TSYS, Debt Manager, Imaging systems,
   internal databases, etc.)
3. Cross-reference values across systems (e.g., compare a balance in TSYS
   against a balance in Debt Manager)
4. Apply business rules (e.g., "if the account was opened before 2019, use
   System B instead of System A")
5. Check timelines (e.g., "was this action taken within 30 days of the event?")
6. Verify document existence (e.g., "is there a Right Party Contact letter
   on file in the imaging system?")
7. Read and interpret free-text agent notes (e.g., "did the agent document
   the customer's reason for dispute?")
8. Record each check as pass/fail in an Excel spreadsheet
9. Submit the completed report for supervisory review and sign-off

This is done every month, for every QC, for every account in the sample.

================================================================================
4. WHY THIS IS A PROBLEM
================================================================================

4.1 SLOW
- Each QC takes significant analyst time every month
- Analysts must navigate multiple systems, often with different interfaces
- Multi-hop lookups are common: to check one thing, you may need to look up
  an account number in System A, use that to find a case ID in System B,
  then use the case ID to find a document in System C

4.2 ERROR-PRONE
- Manual cross-referencing across 15-20 source systems
- Copy-paste errors between systems and spreadsheets
- Fatigue errors on repetitive checks across large samples
- Inconsistent interpretation of ambiguous PnP language

4.3 NOT SCALABLE
- Adding a new QC requires training analysts on a new PnP
- Institutional knowledge lives in people's heads, not in systems
- When an analyst leaves, their tribal knowledge goes with them
- Onboarding a new analyst to a complex QC takes weeks
- WITH 300+ QCs AND A TARGET OF 30-40 NEW QCs PER QUARTER, the manual
  approach simply cannot keep up. This is not a problem that can be
  solved by hiring — the knowledge transfer bottleneck is the constraint,
  not headcount.

4.4 INCONSISTENT
- Different analysts may interpret the same PnP step differently
- "Verify the amount is correct" — correct compared to what? Which system?
  What tolerance? The PnP often doesn't specify.
- Subjective checks (like reading agent notes) vary by analyst judgment

4.5 AUDIT RISK
- Regulators expect consistent, documented, repeatable QC processes
- Manual processes are harder to audit — the "how" lives in people's heads
- Any inconsistency in execution can become a regulatory finding

================================================================================
5. SOURCE SYSTEMS
================================================================================

QC processes pull data from approximately 15-20 internal systems. Examples:

- TSYS: Card processing system (balances, transactions, account status)
- Debt Manager: Collections management (payment plans, account assignments)
- Imaging System: Document storage (letters, notices, correspondence)
- Internal databases: Various operational databases
- Other systems specific to business units

- Excel Trackers (Shared Folders): Many QC processes rely on Excel files
  maintained in shared network folders as DATA SOURCES — not just output
  reports. Analysts pull reference data, lookup values, pre-processed
  extracts, and intermediate calculations from these trackers.

Key characteristics of these systems:
- Each has its own API, interface, or database access method
- Data formats and field names are inconsistent across systems
- Some systems update in real-time, others batch (e.g., TSYS updates daily,
  Debt Manager updates weekly)
- Multi-hop lookups are common (account → case → document)
- Some systems may be legacy with limited API access
- System migrations happen — accounts that were in one system may have been
  migrated to another, and the PnP may not reflect this

Special challenge — Excel trackers as data sources:
- Stored in shared drives/network folders with inconsistent folder structures
- File naming conventions vary by QC, business unit, and even individual analyst
- Updated manually by other teams on varying (and often undocumented) schedules
- No version control — the "current" file is whatever's in the folder
- Column names, formats, sheet names, and layouts differ across trackers
- Some trackers are generated by macros, manual exports, or pivot table refreshes
  that must be run BEFORE the QC process can begin
- An analyst might open Tracker A, look up an account, get a reference number,
  then use that reference to query a formal system — the tracker is a bridge
- These are fragile: if someone moves, renames, restructures, or overwrites
  the file, the process silently breaks
- Some trackers contain derived data (calculations, formulas) that are themselves
  a source of truth for the QC — not just a passthrough
- There is no centralized inventory of which trackers exist, where they live,
  or which QCs depend on them

================================================================================
6. PnP DOCUMENTS — THE SOURCE OF TRUTH (AND ITS PROBLEMS)
================================================================================

PnP (Policy & Procedure) documents are the official instructions for each QC.
They are written by compliance/policy teams and are supposed to tell the QC
analyst exactly what to do.

6.1 FORMAT VARIATION
- Some PnPs are well-structured: numbered steps, clear fields, specific systems
- Some are narrative paragraphs: "The analyst should verify that the balance
  is consistent with the expected recovery amount"
- Some are scanned PDFs (OCR may be needed)
- Some mix all of these formats within a single document
- Length ranges from 2 pages to 30+ pages

6.2 PnP DOCUMENTS HAVE GAPS
This is a critical problem. PnPs are often incomplete or ambiguous:

STRUCTURAL GAPS (what's literally missing from the document):
- Missing data source: "Verify the balance is correct" — from which system?
- Missing field names: "Check the payment date" — which field exactly?
- Missing tolerances: "Amounts should match" — exactly, or within $0.01?
  Within $1.00? Within 1%?
- Missing fallback sources: "If System A is unavailable" — then what?

LOGICAL GAPS (what doesn't make sense):
- Implicit dependencies between steps not stated in the PnP
  (e.g., Step 5 requires the result of Step 3, but this isn't mentioned)
- Ambiguous ordering: does step order matter? Can some be done in parallel?
- Edge cases not addressed: what happens if an account has no transactions?
  If a field is null? If the account was closed mid-period?

DOMAIN-SPECIFIC GAPS (knowledge that exists only in people's heads):
- System migration exceptions: "Migrated accounts use Legacy Table X, not TSYS"
- Regulatory changes: "The RPC (Right Party Contact) definition changed after
  the 2023 regulatory update, but the PnP hasn't been updated yet"
- Data timing constraints: "TSYS updates daily but Debt Manager updates weekly
  — you can only reliably compare them after Thursday"
- Business rule overrides: "Skip this check for accounts under $500"
- Unwritten workarounds: "For accounts opened before 2019, use System B
  instead of System A"
- Seasonal exceptions: "During Q4, the tolerance is relaxed to $5.00 because
  of year-end adjustments"

POLITICAL/ORGANIZATIONAL GAPS (knowledge the AI can never surface):
- Undocumented process changes that were approved verbally but never
  written into the PnP
- Intentional ambiguity left in PnPs for political reasons
- Processes that technically violate the PnP but are accepted practice
- Disagreements between business units about the correct procedure

The implication: you CANNOT fully automate QC by just reading the PnP.
The PnP is necessary but not sufficient. A significant portion of the
knowledge required to execute QC correctly lives in the heads of experienced
analysts (SMEs — Subject Matter Experts).

================================================================================
7. TYPES OF VALIDATION CHECKS
================================================================================

Based on analysis of QC processes, most validation steps fall into a finite
set of check types. Estimated 8-10 types cover the majority:

7.1 CROSS-MATCH NUMERIC
Compare a numeric value from System A against System B.
- Example: "Balance in TSYS should match balance in Debt Manager"
- Edge cases: tolerances ($0.01? $1.00?), null values, currency conversion,
  negative amounts, zero-balance accounts

7.2 CROSS-MATCH DATE
Compare dates from two systems.
- Example: "Account open date in System A should match System B"
- Edge cases: timezone differences, date format mismatches (MM/DD vs DD/MM),
  business days vs calendar days, null dates

7.3 CROSS-MATCH STRING
Compare text values from two systems.
- Example: "Customer name in System A should match System B"
- Edge cases: nickname variations, middle name inclusion, case sensitivity,
  special characters, truncation in one system, encoding differences

7.4 THRESHOLD CHECK
Verify a value is above or below a limit.
- Example: "Account balance should not exceed $50,000"
- Edge cases: boundary values (exactly equal?), dynamic thresholds that
  change based on account type or business unit

7.5 TIMELINE CHECK
Verify an action was taken within N days of an event.
- Example: "First collection attempt must be within 30 days of delinquency"
- Edge cases: business days vs calendar days, holidays, weekends,
  what counts as "day 1"?, accounts that span month boundaries

7.6 DOCUMENT EXISTS
Verify a specific document is on file.
- Example: "Right Party Contact letter must be in the imaging system"
- Edge cases: multiple versions of the document, documents filed under
  wrong account, OCR quality of scanned documents, document naming
  conventions varying by time period

7.7 CONDITIONAL CHECK
If condition X is true, then Y must also be true.
- Example: "If account is in collections, a payment plan must be on file"
- Edge cases: nested conditions, conditions that depend on data from
  multiple systems, conditions based on historical state (what was true
  at the time of the event, not what's true now)

7.8 LIST CONTAINS
Verify a value is in an allowed set.
- Example: "Account status must be one of: Active, Suspended, Under Review"
- Edge cases: status codes that differ between systems, deprecated status
  codes still in old accounts, case sensitivity

7.9 NOTE ANALYSIS (FREE-TEXT)
Read and interpret free-text agent notes to determine if specific actions
were documented.
- Example: "Agent notes must document the customer's reason for dispute"
- This is fundamentally different from the other check types:
  - Cannot be validated deterministically
  - Requires natural language understanding
  - Highly subjective — different analysts may disagree
  - Agent notes vary wildly in quality, length, and format
  - Abbreviations, shorthand, and jargon are common
  - Notes from different time periods may use different terminology
  - A single note field may contain entries from multiple agents over time

================================================================================
8. EDGE CASES AND COMPLICATIONS
================================================================================

8.1 MULTI-HOP DATA LOOKUPS
Many checks require chaining across systems:
- Look up account number → get case ID → get document reference → verify
  document exists in imaging system
- If any hop fails (system unavailable, ID not found, data mismatch), the
  entire check cannot be completed

8.2 TEMPORAL DEPENDENCIES
- Some data is only valid at certain times (weekly batch systems)
- Historical state matters: "Was the account in good standing AT THE TIME
  the decision was made?" — not whether it is now
- Retroactive adjustments: data may change after the fact

8.3 ACCOUNT LIFECYCLE EDGE CASES
- Accounts that were opened, closed, and reopened
- Accounts that migrated between systems mid-lifecycle
- Accounts involved in mergers or acquisitions
- Joint accounts vs individual accounts
- Deceased account holders
- Accounts under legal hold or litigation

8.4 REGULATORY VARIATION
- Different QCs may be governed by different regulations
- Regulations change over time, but PnPs may lag behind
- State-level vs federal regulations may conflict
- Some checks apply only to specific account types or geographies

8.5 SAMPLE SELECTION
- Not all accounts are checked — only a sample
- Sample selection methods vary by QC (random, risk-based, stratified)
- The automation system needs to handle whatever accounts are in the sample
- Some accounts in the sample may be unusual edge cases

8.6 QC INTERDEPENDENCIES
- Some QC processes overlap — the same account may be checked by multiple QCs
- Results from one QC may affect the interpretation of another
- Some QCs share common sub-checks

8.8 PROCESS-LEVEL HETEROGENEITY
Each QC process is essentially its own mini-world. The differences between
QC processes go far beyond just different data sources:

- DIFFERENT DATA SOURCES: QC #1 might use TSYS + Debt Manager. QC #2 might
  use Imaging + an Excel tracker from a shared folder. QC #3 might use an
  entirely different set of systems. There is no common denominator.

- DIFFERENT PROCEDURES AND LOGIC: The execution flow itself differs — one QC
  might be a simple sequential checklist, another might have complex branching
  logic ("if X, check Y; if not X, skip to step 12 and check Z instead").
  The procedures are not just different steps — they are different SHAPES of
  processes.

- DIFFERENT STEP TYPES: Not just different counts (5 vs 50 steps), but
  fundamentally different types of work — one QC might be all numeric
  cross-matches, another might be mostly document verification and note
  analysis, another might be entirely timeline-based.

- DIFFERENT TOOLING: Some QCs rely on analyst-maintained Excel trackers stored
  in shared folders as lookup tables, pre-processed extracts, or intermediate
  data. These trackers vary in location, format, column naming, update
  frequency, and ownership. Some QCs require no trackers at all.

- DIFFERENT PRE-PROCESSING: Some QCs require the analyst to first generate or
  refresh data (run a macro, export a report from a system, refresh a pivot
  table, pull a new extract) before the actual QC steps can even begin. The
  PnP may or may not document these pre-processing steps.

- DIFFERENT TERMINOLOGY: The same concept may be called different things across
  business units (e.g., "payment plan" vs "arrangement" vs "workout agreement"
  vs "repayment schedule" — all meaning the same thing in different QCs).

- DIFFERENT EXCEPTION HANDLING: What counts as an acceptable deviation or
  exception varies by QC, by business unit, and sometimes by analyst.

The implication: no single template, script, or hardcoded workflow will work
for all QCs. The system must be flexible enough to handle this diversity while
still being standardized enough to scale. This tension — between per-QC
flexibility and cross-QC standardization — is one of the core design challenges.

8.9 HUMAN JUDGMENT AREAS
Certain checks inherently require human judgment and cannot be fully automated:
- Interpreting ambiguous agent notes
- Determining "reasonableness" of an action
- Evaluating whether an exception was justified
- Assessing the adequacy of documentation
These areas need a mechanism to surface them for human review rather than
attempting to automate the judgment itself.

================================================================================
9. STAKEHOLDERS AND THEIR CONCERNS
================================================================================

9.1 QC ANALYSTS (SMEs)
- Primary users of the current manual process
- Hold tribal knowledge about system quirks and PnP gaps
- Concern: will this replace their jobs?
- Need: tools that reduce tedious work, not tools that bypass them
- They are the essential source of domain knowledge during onboarding

9.2 QC MANAGERS / SUPERVISORS
- Review and sign off on QC results
- Need: consistent, auditable results they can trust
- Concern: accountability — who is responsible if automation makes an error?

9.3 COMPLIANCE / RISK
- Need: full audit trail of every check performed
- Need: evidence that the same process is followed every time
- Concern: regulatory defensibility — can we explain to a regulator
  exactly how each check was performed?

9.4 CDO / EXECUTIVE LEADERSHIP
- Need: scalability (300+ QCs), efficiency, cost reduction
- Need: de-risked approach (experimentation before full build)
- Concern: timeline, cost, and whether it actually works
- Concern: how does this compare to just hiring more analysts?

9.5 ENGINEERING / TECHNOLOGY
- Will build and maintain the system
- Need: clear requirements, extensible architecture
- Concern: integration with legacy systems, data access, LLM reliability

================================================================================
10. SME DEPENDENCY — WHAT THE PROJECT NEEDS FROM SUBJECT MATTER EXPERTS
================================================================================

SMEs (QC analysts with deep domain experience) are not just stakeholders — they
are an essential, irreplaceable input to the project. Without active SME
participation, the system cannot be built. This section catalogs every type of
help required from SMEs so the project can plan for it.

10.1 PnP INTERPRETATION AND PROCESSING
- Translating ambiguous PnP language into specific, executable validation steps
- Clarifying which systems the PnP actually means ("the system" → which one?)
- Identifying which fields to use ("check the balance" → which balance field?)
- Specifying thresholds and tolerances the PnP leaves vague
- Flagging steps that are implicit — things analysts always do but the PnP
  doesn't mention ("everyone knows you have to check the migration flag first")
- Identifying which Excel trackers are needed, where they live, what columns
  to use, and how often they're updated
- Explaining the SEQUENCE and DEPENDENCIES between steps when the PnP doesn't
  make them clear

10.2 GAP IDENTIFICATION AND RESOLUTION
- Validating AI-detected gaps ("yes, the PnP really is missing the source system
  for this step — it should be TSYS")
- Filling gaps with tribal knowledge that exists nowhere in writing
- Providing precedents ("we had this same gap in another QC — here's how we
  resolved it")
- Identifying gaps the AI MISSED — things the AI accepted as complete but an
  experienced analyst would flag as insufficient
- Documenting unwritten rules, workarounds, seasonal exceptions, and system
  quirks that the PnP doesn't capture

10.3 EVALUATION AND VALIDATION
- Reviewing AI-generated configurations against their expert understanding
  of how the QC should actually work
- Providing representative sample accounts for testing — including "normal"
  accounts AND known edge cases
- Identifying the "hard" accounts that will stress-test the system
- Comparing automated results against their manual results (parallel runs)
- Confirming PASS/FAIL results match expected outcomes for known accounts
- Validating that the system handles edge cases correctly (migrated accounts,
  closed-reopened accounts, null values, etc.)

10.4 EXPERIMENTATION SUPPORT (PHASE 0)
- Participating in early-phase experimentation to validate whether the
  AI can even parse their specific PnP documents
- Providing feedback on where the AI gets it right vs. wrong, with
  explanations of WHY it's wrong (not just "that's incorrect")
- Selecting 2-3 representative QCs for the pilot — ideally ones that cover
  a range of complexity (one simple, one moderate, one complex)
- Being available for rapid iteration during experimentation — answering
  questions, reviewing outputs, correcting misinterpretations

10.5 REFINEMENT AND CORRECTION
- Correcting automated results when they're wrong — and explaining the
  reasoning ("this is wrong because migrated accounts use a different table,
  not because the logic is flawed")
- Feeding corrections back to the system so it learns and doesn't repeat
  the same mistakes (building the Correction Store)
- Iterating on configurations after the first run — reviewing, adjusting
  thresholds, adding missed conditions, refining logic
- Distinguishing between "the system got this wrong" vs. "the PnP is
  actually wrong/outdated" — both happen, but the fix is different

10.6 DOMAIN KNOWLEDGE TRANSFER
- Documenting tribal knowledge that currently lives only in their heads
- Explaining system quirks and data timing constraints ("TSYS updates daily
  at 2 AM, but Debt Manager updates weekly on Thursdays — don't compare
  them on Monday")
- Explaining business context that the AI needs to interpret PnPs correctly
  (what a "workout agreement" is, how "Right Party Contact" rules changed
  after the 2023 regulatory update, etc.)
- Sharing knowledge about which Excel trackers exist, where they're stored,
  who maintains them, and what their update schedules are

10.7 ONGOING MAINTENANCE AND MONITORING
- Reviewing results when PnPs are updated by compliance/policy teams
- Flagging when source system changes affect QC processes (field renames,
  system migrations, new data sources)
- Providing corrections when the system drifts (regulatory changes,
  new business rules, seasonal adjustments)
- Spot-checking automated results periodically to maintain confidence

10.8 THE SME AVAILABILITY CONSTRAINT
- SMEs have their regular QC work to do — they cannot dedicate full time
  to the project
- Different QCs may have different SMEs — there is no single person who
  knows all 300+ QCs
- Some QCs may have only ONE knowledgeable SME — if that person is
  unavailable, onboarding that QC stalls
- SME engagement must be structured to minimize their time burden while
  maximizing knowledge capture
- Every interaction with an SME should be designed to extract and
  permanently capture knowledge, so the same question never needs to
  be asked twice

The implication: the system's ability to scale is directly constrained by
how efficiently it uses SME time. If onboarding each QC requires 40 hours
of SME time, the 30-40 QCs/quarter target is impossible. The system must
progressively reduce SME time per QC as it accumulates knowledge.

11. CONSTRAINTS
================================================================================

11.1 REGULATORY / COMPLIANCE
- Full audit trail required for every automated check
- Results must be explainable — "the AI said so" is not acceptable
- No automated production actions (no closing QCs, no updating records)
- Human sign-off required on all results
- Must be at least as accurate as the manual process
- THIS PROJECT HAS HIGH VISIBILITY FROM A RISK PERSPECTIVE. QC is a
  compliance function — errors in QC can lead to regulatory findings,
  fines, and reputational damage. Accuracy is not optional. The system
  must be demonstrably reliable before being trusted with production QCs.
  Any automated result that is wrong is worse than a manual result that
  is slow — because it creates a false sense of compliance.

11.2 TECHNICAL
- 15-20 source systems with varying access methods
- Some systems are legacy with limited or no API access
- Data may need to be staged in a data lake before querying
- LLM accuracy against real PnP documents is unknown until tested
- System availability and latency vary across source systems
- THE AUTOMATION MUST NOT OVERLOAD PRODUCTION SYSTEMS. When running QC
  across hundreds of accounts, the system will make thousands of data
  requests. Hitting production databases or APIs directly — especially
  repeatedly or in parallel — risks degrading or bringing down critical
  bank systems that serve live customers. The design must account for
  this: either stage/cache data before QC runs, batch requests during
  off-peak hours, or use read replicas. This is a non-negotiable
  constraint — the automation cannot become a threat to system stability.
- AUTHENTICATION AND DATA ACCESS MUST BE CENTRALIZED. Agents should NOT
  hold credentials or connect directly to source systems. There must be
  a centralized data layer that handles all authentication, authorization,
  and data retrieval. Agents request data from this layer based on their
  assigned roles/permissions — they never touch source systems directly.
  This ensures: (1) credential management is in one place, not scattered
  across agents, (2) access control is enforced consistently (an agent
  running a collections QC should not be able to query fraud systems),
  (3) all data access is logged and auditable through a single gateway,
  (4) adding or rotating credentials doesn't require changing agent code.

11.3 ORGANIZATIONAL
- SME time is limited — they have to do their regular QC work too
- PnP documents are owned by compliance/policy teams, not QC teams
- Getting PnPs updated is a slow process (months)
- Different business units may have different priorities and timelines
- Change management — analysts need to trust the system before relying on it

11.4 SCALE — THE DEFINING CONSTRAINT
- THIS IS THE MOST IMPORTANT CONSTRAINT IN THE ENTIRE PROJECT.
- Without scalability, this is just another automation project — one team
  building bespoke scripts for one QC at a time. That approach cannot
  cover 300+ QCs in any reasonable timeframe.
- Target: onboard 30-40 QCs per quarter once the system is production-ready
- 300+ total QCs means multi-year rollout even at 30-40/quarter
- The system must get better over time, not stay static
- Adding a new QC should ideally require zero code changes — just configuration
- NO BESPOKE AGENTS. PERIOD.
  The system CANNOT have a custom-built agent, pipeline, or workflow
  for each QC. With 300+ QCs, that means 300+ things to build, test,
  maintain, debug, and update — every time a regulation changes, every
  time a source system migrates, every time a PnP is revised. It is
  unmaintainable at scale. The entire point of the architecture is ONE
  generic, configurable system that can handle ANY QC by reading its PnP
  and producing a configuration. The same engine, the same agent logic,
  the same execution pipeline — driven by different configurations, not
  different code. If the design drifts toward "let's build a special
  agent for credit card QCs and another for collections QCs and another
  for fraud QCs," it has failed. The differences between QCs must live
  in DATA (configurations, rules, connector mappings) — never in CODE.
- WHAT MAKES THIS AGENTIC (NOT JUST AUTOMATION):
  The system must be able to read a new PnP document it has never seen,
  understand the validation steps, map them to existing rules, identify
  gaps, and produce a working configuration — with minimal human input.
  If every new QC requires weeks of engineering work, the economics don't
  work and leadership will (rightly) question why we didn't just hire
  more analysts.
- The onboarding cost per QC must DECREASE over time as the system learns
  from past QCs (via accumulated corrections, gap patterns, and domain
  knowledge). QC #1 will be hard. QC #50 should be significantly faster.
  QC #200 should be near-automatic.
- This scaling property is what justifies the upfront investment in an
  agentic architecture. Without it, a simpler approach would suffice.

11.5 DATA SENSITIVITY
- Financial data is highly sensitive (PII, account numbers, balances)
- LLMs must not be trained on or retain customer data
- Data must stay within the institution's security perimeter
- Cloud LLM usage must comply with the institution's data governance policies

================================================================================
12. WHAT SUCCESS LOOKS LIKE
================================================================================

- A QC analyst's monthly work is reduced from hours/days to reviewing a
  pre-filled report and making judgment calls on ambiguous items
- Adding a new QC process requires an SME to spend a few hours with the
  system (not weeks of training), after which it runs automatically
- The system improves with every QC onboarded — knowledge accumulates
- Audit trail is automatic and complete
- Human judgment is preserved for cases that genuinely need it
- The system says "I don't know" (flags for human review) rather than
  guessing when it's uncertain
- No QC is ever closed or finalized without human sign-off
- CRITICALLY: onboarding a new QC takes hours (not weeks), and the effort
  per QC decreases over time as the system accumulates domain knowledge.
  By QC #50+, most onboardings are near-automatic with SME confirmation.
  This is what makes the 30-40 QCs/quarter target achievable.

================================================================================
13. WHAT THIS IS NOT
================================================================================

- This is NOT an autonomous system that replaces QC analysts
- This is NOT a system that takes production actions (close accounts, etc.)
- This is NOT a chatbot or conversational interface
- This is NOT a rules engine that someone manually programs rule by rule
- This is NOT 300+ bespoke agents — it is ONE configurable system that
  handles any QC through configuration, not custom code. The differences
  between QCs live in data (configs, rules, mappings), never in code.
- This is an automation system that executes the checks and surfaces results
  for human review — automation of execution, not automation of judgment
